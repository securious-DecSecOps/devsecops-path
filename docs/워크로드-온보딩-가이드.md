# 워크로드 온보딩 가이드

새 워크로드(예: KubeGoat, OWASP Juice Shop 등)를 이 PoC의 Golden Path 위에 태우려는 사람을 위한 contract 명세 + 절차서.

기준: tag v0.3.2-poc, Build #21 검증 시점.
대상 독자: 이 PoC를 처음 받아서 자기 워크로드를 올리려는 팀원, 본인이 향후 다른 워크로드 추가 시 참고용.

---

## 0. 먼저 읽을 것

이 PoC는 **VulnBank MSA를 reference implementation으로 만든 Golden Path**다.
파이프라인 자체는 환경 변수로 파라미터화되어 있지만, default 값과 일부 스크립트가 VulnBank 패턴을 전제한다.
즉 새 워크로드는 다음 둘 중 하나가 필요하다.

- (A) **본 contract를 따르는 형태로 새 파일 N개를 만든다.** 스크립트 수정 없이 새 Jenkins job 1개만 생성하면 동작. 권장.
- (B) **기존 스크립트의 하드코딩 부분을 환경 변수로 추출해서 generic화한다.** 작업량이 더 크지만 진짜 generic platform으로 진화. 다음 단계.

이 문서는 (A) 기준이다. (B)에 필요한 리팩토링 후보는 8장에 모아두었다.

---

## 1. Golden Path가 보장하는 것

새 워크로드가 contract만 따르면 다음이 자동으로 돈다.

- **Build**: 모든 service의 Docker image 빌드
- **Scan**: 5종 정적 도구 + SBOM 생성 (Trivy / SBOM / Kubescape / Checkov / Gitleaks / SonarQube)
- **Gate**: 정책에 따라 빌드 차단 또는 정보 기록 (ENFORCE_GATE 토글)
- **Push**: 모든 이미지가 Harbor에 push
- **GitOps commit**: gitops-manifest-repo에 이미지 tag bump가 자동 commit/push
- **Deploy**: ArgoCD가 새 commit 감지 후 sync, 새 Pod 기동
- **Evidence**: `reports/dev/<BUILD_NUMBER>/` 디렉토리에 모든 결과 archive, Jenkins archive에 영구 보존

여기서 보장하지 않는 것:
- **워크로드 특화 e2e test**(예: VulnBank의 `verify.sh` 같은 의도된 vuln 재현 검증) — 워크로드마다 별도 작성 필요
- **워크로드별 보안 정책** — SonarQube quality gate, Kubescape framework 선택, Checkov skip rule 등은 default 적용. 워크로드별로 분리하려면 정책 파일 분리 필요

## 2. Contract — 새 워크로드가 제공해야 하는 것

3-repo 각각에 필요한 산출물을 정리한다.

### 2.1 app-source-repo 측

```
app-source-repo/
└── examples/<workload-name>/
    └── services/
        ├── <svc1>/
        │   ├── Dockerfile         (필수, build context = 이 디렉토리)
        │   └── public/            (선택, 앱 코드)
        │       └── openapi.json   (선택, OpenAPI 사용 시)
        ├── <svc2>/
        └── ...
```

**규약**:

- 서비스 디렉토리 이름이 곧 service name (`SERVICES` 환경 변수에 들어감)
- 각 service는 `docker build .`로 그 디렉토리에서 빌드 가능해야 함
- 빌드된 이미지 tag pattern: `${REGISTRY_URL}/${REGISTRY_PROJECT}/${APP_NAME}-<svc>:${IMAGE_TAG}`
  - 예: `localhost:8082/secubank/vulnbank-msa-frontend:21`
  - 새 워크로드: `localhost:8082/secubank/kubegoat-webapp:1` 같은 패턴
- service 간 통신은 K8s Service DNS 사용 권장 (현재 PoC는 namespace=`secure-path-dev`)

**single-service 워크로드도 가능**: `SERVICES=webapp` 같은 식으로 1개만 등록해도 됨. 6 service가 아닌 N개에 일반화 가능.

### 2.2 gitops-manifest-repo 측

```
gitops-manifest-repo/
├── helm/<workload-name>/
│   ├── Chart.yaml
│   ├── values.yaml                  (default values)
│   └── templates/
│       ├── deployment.yaml
│       ├── service.yaml
│       └── ...
├── apps/<workload-name>/<env>/
│   ├── values.yaml                  (env overlay — 이미지 tag가 여기 들어감)
│   └── kustomization.yaml           (선택)
└── argocd/applications/
    └── <workload-name>-<env>.yaml   (ArgoCD App 정의)
```

**규약**:

- Helm chart 위치: `helm/<workload-name>/` (Jenkinsfile의 `HELM_CHART_DIR` 파라미터로 지정)
- 환경 overlay 위치: `apps/<workload-name>/<env>/values.yaml` (Jenkinsfile의 `GITOPS_APP_DIR` 파라미터)
- **이미지 tag 위치는 일관되어야** Jenkins가 자동 bump 가능. VulnBank 패턴 예시:
  ```yaml
  services:
    <svc>:
      image:
        repository: localhost:8082/secubank/<APP>-<svc>
        tag: "<BUILD_NUMBER>"
  ```
- ArgoCD App 이름: `secubank-<workload-name>-<env>` (Jenkinsfile의 `ARGOCD_APP_NAME` 파라미터)

**chart 자체는 자유**. multi-service Helm chart, plain manifest + Kustomize, 외부 Helm chart의 values overlay 모두 가능. 단, 이미지 tag가 들어가는 위치만 일관되면 됨.

### 2.3 devsecops-path 측

Contract 측면에서는 **수정 없음**. 다음 둘 중 하나만 추가하면 됨:

- (방법 1, 권장) **새 Jenkins job 1개 생성** (workload별로 분리)
- (방법 2) 기존 `vulnbank-msa-dev` job에서 build trigger 시 모든 파라미터 override

방법 1이 더 명확하고 다른 workload와 격리됨.

## 3. 어디를 만지는가 — 3-repo 변경 매트릭스

새 workload 온보딩 시 만지는 / 안 만지는 파일을 한눈에:

| Repo | 만지는 것 | 안 만지는 것 |
| --- | --- | --- |
| `app-source-repo` | `examples/<workload-name>/services/*/` 신규 | 기존 `examples/vulnbank-msa/` 그대로 |
| `gitops-manifest-repo` | `helm/<workload-name>/`, `apps/<workload-name>/dev/`, `argocd/applications/<workload-name>-dev.yaml` 신규 | 기존 `helm/vulnbank-msa/` 그대로 |
| `devsecops-path` | Jenkins job 1개 생성 (bootstrap script 사용) | Jenkinsfile.msa, scripts/*, bootstrap/local-wsl/* 모두 그대로 |

→ 이상적으로는 **devsecops-path repo에 commit 0건**. 새 workload 추가가 인프라 repo를 안 건드리는 게 Golden Path의 미덕.

현실: 아래 8장의 하드코딩들 때문에 일부 스크립트는 새 workload용으로 fork하거나 환경 변수 추가 필요.

## 4. 단계별 온보딩 절차 (10 step)

새 workload 이름을 `myworkload`라 가정한다. 실제로는 `kubegoat`, `juice-shop` 등.

### Step 1 — app-source-repo에 서비스 추가

```bash
cd /home/wngus/app-source-repo
mkdir -p examples/myworkload/services/webapp
# 본인 워크로드의 Dockerfile + 소스 복사
cp -r ~/source/myworkload-src examples/myworkload/services/webapp/
# Dockerfile 위치 확인: examples/myworkload/services/webapp/Dockerfile
```

### Step 2 — 로컬에서 Dockerfile이 동작하는지 확인

```bash
cd examples/myworkload/services/webapp
docker build -t test .   # 통과해야 다음 단계 가능
```

### Step 3 — gitops-manifest-repo에 Helm chart 작성

```bash
cd /home/wngus/gitops-manifest-repo
mkdir -p helm/myworkload/templates
# Chart.yaml, values.yaml, templates/deployment.yaml, templates/service.yaml 작성
# 이미지 tag가 들어갈 위치를 values.yaml의 일관된 path에 둘 것
```

`helm template myworkload helm/myworkload/`로 렌더링이 되는지 확인.

### Step 4 — 환경 overlay 작성

```bash
mkdir -p apps/myworkload/dev
cat > apps/myworkload/dev/values.yaml <<EOF
services:
  webapp:
    image:
      repository: localhost:8082/secubank/myworkload-webapp
      tag: "dev"
EOF
```

`tag: "dev"`가 초기 값. Jenkins가 build마다 BUILD_NUMBER로 자동 bump할 것.

### Step 5 — ArgoCD Application 작성

```bash
mkdir -p argocd/applications
cat > argocd/applications/myworkload-dev.yaml <<EOF
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: secubank-myworkload-dev
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/securious-DecSecOps/gitops-manifest-repo.git
    targetRevision: main
    path: helm/myworkload
    helm:
      valueFiles:
        - ../../apps/myworkload/dev/values.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: myworkload-dev
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
EOF
```

### Step 6 — gitops-manifest-repo commit + push

```bash
git add helm/myworkload apps/myworkload argocd/applications/myworkload-dev.yaml
git commit -m "myworkload Helm chart + ArgoCD App 추가 완료"
git push origin main
```

### Step 7 — Jenkins job 생성

기존 bootstrap script를 환경 변수 override해서 재사용:

```bash
JENKINS_JOB_NAME=myworkload-dev \
WORKLOAD_NAME=myworkload \
APP_NAME=myworkload \
NAMESPACE=myworkload-dev \
HELM_CHART_DIR=gitops-manifest-repo/helm/myworkload \
GITOPS_APP_DIR=gitops-manifest-repo/apps/myworkload/dev \
ARGOCD_APP_MANIFEST=gitops-manifest-repo/argocd/applications/myworkload-dev.yaml \
ARGOCD_APP_NAME=secubank-myworkload-dev \
MSA_WORKLOAD_DIR=app-source-repo/examples/myworkload \
SERVICES=webapp \
HELM_RELEASE=myworkload \
JENKINS_USER=admin \
JENKINS_TOKEN=<your-token> \
  bash /home/wngus/devsecops-path/bootstrap/local-wsl/30-jenkins-job.sh
```

→ Jenkins에 `myworkload-dev` job이 생성됨. SBOM, Kubescape, Trivy 등 모든 stage가 myworkload-specific 값으로 동작.

### Step 8 — 자격 증명 공유 확인

기존 Jenkins credential (Harbor robot, GitHub PAT, SonarQube token)을 그대로 사용 가능. 단, SonarQube project key를 분리하려면 `SONAR_PROJECT_KEY=myworkload`로 trigger 시 override (또는 sonarqube-scan-services.sh의 default 수정).

### Step 9 — 첫 빌드 트리거

```bash
JENKINS_USER=admin JENKINS_TOKEN=<token> \
JENKINS_JOB_NAME=myworkload-dev \
  bash /home/wngus/devsecops-path/bootstrap/local-wsl/50-trigger-build.sh
```

→ 16 stage 진행. Harbor에 image push, gitops repo에 자동 commit, ArgoCD가 sync.

### Step 10 — 검증

```bash
kubectl get pods -n myworkload-dev
kubectl -n argocd get application secubank-myworkload-dev
# Synced + Healthy 확인

# evidence
docker exec secubank-jenkins ls /var/jenkins_home/jobs/myworkload-dev/builds/1/archive/reports/dev/1/
```

## 5. 환경 변수 vs 하드코딩 — 현재 상태

contract를 어디까지 따르면 스크립트 수정 없이 동작하는지 정확히:

### 5.1 환경 변수만으로 해결 가능 (대부분)

- 모든 Jenkinsfile.msa의 파라미터 (WORKLOAD_NAME, APP_NAME, SERVICES, NAMESPACE, REGISTRY_*, HELM_*, GITOPS_*, ARGOCD_*, SONAR_*)
- `scripts/build-services.sh`, `scripts/trivy-scan-services.sh`, `scripts/generate-sbom.sh`, `scripts/kubescape-scan.sh`, `scripts/checkov-scan-services.sh`, `scripts/gitleaks-scan-repos.sh` — 모두 env 기반
- `scripts/push-services.sh` — env 기반
- `scripts/deploy-argocd.sh` — env 기반
- `scripts/collect-msa-evidence.sh`의 일부 — env 기반

### 5.2 하드코딩이 남아 있어 fork/패치가 필요한 부분

| 파일 | 하드코딩 위치 | 회피 방법 |
| --- | --- | --- |
| `scripts/gitops-commit-push.sh:59,85,88` | `apps/vulnbank-msa/dev/values.yaml` 경로 | `${GITOPS_APP_DIR}/values.yaml`로 추출 필요 OR 워크로드별 fork |
| `scripts/gitops-commit-push.sh:50` | commit author `Jenkins CI (vulnbank-msa)` | env `GITOPS_COMMIT_NAME` 이미 있음. trigger 시 override |
| `scripts/checkov-scan-services.sh:111` | `helm template vulnbank-msa` (release 이름) | `${HELM_RELEASE}`로 추출 필요 |
| `scripts/sonarqube-scan-services.sh:5` | `SONAR_PROJECT_KEY` default | trigger 시 `SONAR_PROJECT_KEY=<workload>` override 가능 |
| `scripts/collect-msa-evidence.sh` | `/vulnbank/online/` 경로, 의도된 vuln 재현 시도 | 새 워크로드에는 collect 영역만 쓰고 integration test는 skip하거나 워크로드별 스크립트 분리 |
| `scripts/test-msa-integration.sh` | 전체가 VulnBank-specific e2e test | 새 워크로드는 자기 verify 스크립트 작성 |
| `scripts/deploy-kubectl.sh:42` | namespace 치환 `secure-path-dev → ${NAMESPACE}` | 동작 자체는 OK, 패턴이 vulnbank 가정. 검토 필요 |
| `bootstrap/local-wsl/verify.sh` | VulnBank 4 vuln 재현 | 새 워크로드는 자기 verify 작성 |

### 5.3 가벼운 리팩토링으로 generic화 가능한 후보

다음은 1줄~5줄 수정으로 환경 변수화 가능. 다음 PoC 단계로 권장:

1. `gitops-commit-push.sh`의 `apps/vulnbank-msa/dev/values.yaml` → `${GITOPS_APP_DIR#gitops-manifest-repo/}/values.yaml`
2. `checkov-scan-services.sh`의 `helm template vulnbank-msa` → `helm template "${HELM_RELEASE}"`
3. `gitops-commit-push.sh`의 commit message에 `${WORKLOAD_NAME}` 사용

이걸 하면 위 표의 ⚠️ 절반이 자동으로 해소된다.

## 6. 워크로드별 deviation 처리

각 워크로드의 특수성에 대한 대응 패턴:

### 6.1 SonarQube quality gate를 워크로드별로 다르게

trigger 시:
```bash
SONAR_PROJECT_KEY=myworkload \
SONAR_PROJECT_NAME=MyWorkload \
  bash bootstrap/local-wsl/50-trigger-build.sh
```

SonarQube UI에서 quality gate를 project별로 다르게 설정. SAST 결과가 워크로드별로 분리 보존됨.

### 6.2 Kubescape framework 선택을 워크로드별로 다르게

현재 `scripts/kubescape-scan.sh`는 NSA/MITRE/CIS 3개를 항상 돌림.
워크로드별 분기가 필요하면 스크립트에 `KUBESCAPE_FRAMEWORKS` env 변수를 도입해서 list로 override 가능하게 확장 (다음 단계 후보).

### 6.3 빌드 없이 upstream image만 deploy (KubeGoat 같은 경우)

KubeGoat은 Docker Hub에 이미 image가 있고 자체 빌드 의미가 없음.
이 경우 **Jenkins build를 우회**하고 gitops-manifest-repo에 upstream image 직접 참조하는 Helm chart만 만든 후 ArgoCD가 sync. CI 파이프라인은 fork된 자체 빌드가 의미 있을 때만 적용.

대신 배포 후 Kubescape + Falco + verify 같은 runtime 도구는 정상적으로 작동.

### 6.4 e2e test (VulnBank의 verify.sh에 해당)

새 워크로드는 자기만의 의도된 vuln 또는 정상 동작 검증 스크립트를 작성. 위치 권장:

```
devsecops-path/bootstrap/local-wsl/verify-<workload-name>.sh
```

호출 시점은 `Collect Evidence` stage 이후. Jenkinsfile에 stage 추가 또는 별도 cron으로 실행.

## 7. 예시 walk-through — KubeGoat 가상 온보딩

KubeGoat은 multi-component K8s 취약 워크로드. 약식 절차:

### 7.1 빌드 없이 upstream image 사용하는 경우 (~30분)

1. `gitops-manifest-repo/helm/kubegoat/` 작성
   - 또는 [upstream chart](https://github.com/madhuakula/kubernetes-goat) 를 fork해서 values overlay
2. `gitops-manifest-repo/apps/kubegoat/dev/values.yaml` — namespace `kubegoat-dev`, image는 upstream
3. `gitops-manifest-repo/argocd/applications/kubegoat-dev.yaml` — ArgoCD App
4. NetworkPolicy 추가 — 외부 노출 차단 + 다른 namespace와 격리
5. gitops repo commit + push → ArgoCD가 자동 deploy
6. `kubescape scan --include-namespaces kubegoat-dev` — runtime cluster posture 평가
7. `kubectl apply -f kube-bench-job.yaml` — 노드 CIS 평가
8. 발표 시: "VulnBank=앱 layer, KubeGoat=cluster posture. 두 워크로드가 서로 다른 layer를 demo"

이 경우 devsecops-path의 Jenkins job 생성 안 함. Jenkins pipeline은 건너뜀.

### 7.2 풀 빌드 파이프라인까지 적용하는 경우 (~1~2일)

1. `app-source-repo/examples/kubegoat/services/*/Dockerfile` (upstream을 fork해서 자체 빌드 가능하게)
2. gitops-manifest-repo는 7.1과 동일하지만 image repository를 자체 Harbor 경로로
3. Jenkins job 생성 (4장 Step 7~9 절차 그대로)
4. 5장의 하드코딩 회피 필요 — `gitops-commit-push.sh` fork 또는 리팩토링
5. workload별 verify-kubegoat.sh 작성

7.1이 발표 시연 목적에 충분. 7.2는 PoC를 multi-workload 플랫폼으로 격상할 때.

## 8. 알려진 한계 (현 PoC scope)

정직하게 명시. 다음 PoC 단계 자리.

### 8.1 진짜 generic화에 필요한 리팩토링 후보

- `scripts/gitops-commit-push.sh` — 워크로드별 values.yaml 경로 / commit message 동적화
- `scripts/checkov-scan-services.sh` — `helm template` release 이름 환경 변수화
- `scripts/sonarqube-scan-services.sh` — project key default 제거 (env required로)
- `scripts/test-msa-integration.sh`, `scripts/collect-msa-evidence.sh`의 integration test 부분 — 워크로드별 스크립트로 분리

### 8.2 현재 안 되는 것

- 한 Jenkins job에서 multi-workload 빌드 — workload별 job 분리 필요
- workload-specific 보안 정책 자동 적용 — 현재 모든 워크로드가 같은 default 정책
- workload별 evidence 디렉토리 분리 — `reports/dev/<N>/`은 BUILD_NUMBER 기준이라 job 분리하면 자연스럽게 분리되지만, 같은 Jenkins job에서 multi-workload는 안 됨

### 8.3 향후 자리만 잡혀 있는 것

- Falco / Cilium / Istio (런타임 보안) — `gitops-manifest-repo/platform/` 폴더 자리 확정, 미설치
- OWASP ZAP CronJob (DAST) — `gitops-manifest-repo/platform/zap/` 자리 확정, 미설치

## 9. 자주 묻는 질문

**Q. 새 워크로드 추가하려면 devsecops-path repo도 만져야 하나?**
이상적으로는 안 만지는 게 목표지만, 5장의 하드코딩 5~6개 때문에 일부 스크립트 환경 변수 override 또는 가벼운 패치가 필요할 수 있다. 풀 generic은 다음 단계.

**Q. KubeGoat 같은 거 올리려면 Jenkins build를 꼭 거쳐야 하나?**
아니다. 6.3 + 7.1처럼 빌드 우회하고 ArgoCD-only로 올려도 된다. 보안 도구 테스트 목적이면 충분.

**Q. SonarQube finding이 워크로드 간 섞이는 게 문제 아닌가?**
trigger 시 `SONAR_PROJECT_KEY`를 워크로드별로 다르게 override하면 SonarQube가 project를 분리해서 저장. UI에서 project별 dashboard가 분리됨.

**Q. ArgoCD App에 prune: true 두면 위험하지 않나?**
PoC는 dev 환경이라 OK. prod는 prune=false + 수동 sync 권장.

**Q. workload별 NetworkPolicy / RBAC는 누가 책임지나?**
Helm chart 안에 같이 포함하거나, `gitops-manifest-repo/platform/<workload>/networkpolicies/`로 분리. 워크로드 owner의 책임.

**Q. 새 워크로드가 의도된 vuln 없는 일반 앱이면?**
verify 스크립트는 vuln 재현이 아니라 "정상 동작 검증"으로 작성하면 된다. 예: `curl <endpoint>/health` returns 200.
